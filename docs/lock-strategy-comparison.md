# Lock 방식 비교 — 낙관적 락 · 비관적 락 · Redis 분산 락

> 상위 추적 이슈: [#5](https://github.com/team-11st-chat/11th-street/issues/5) · 실행 이슈: [#30](https://github.com/team-11st-chat/11th-street/issues/30)
> 근거: [wiki/Policies#redis-분산-락을-이용한-동시성-제어-정책](https://github.com/team-11st-chat/11th-street/wiki/Policies#redis-분산-락을-이용한-동시성-제어-정책)

이 문서는 타임세일 주문과 쿠폰 발급의 동시성 제어를 위해 검토한 세 가지 락 방식(낙관적 락, 비관적 락, Redis 분산 락)의 장단점과, 최종적으로 **Redisson Pub/Sub 기반 분산 락**을 선택한 근거를 정리한다.

## 1. 해결하려는 문제

타임세일과 선착순 쿠폰은 **한정 수량을 다수의 동시 요청이 경쟁**하는 구조다. 보호하지 않으면 다음 정합성 위반이 발생한다.

- **초과 판매 / 초과 발급**: `잔여 수량 > 0` 검증과 수량 차감 사이의 경합으로, 차감이 직렬화되지 않으면 0 이하로 내려간다.
- **중복 발급**: 동일 고객의 동시 요청이 "이미 발급함" 검증을 동시에 통과해 2건 이상 발급된다.

즉, "**조회 → 검증 → 차감/생성**"이라는 read-modify-write 구간을 원자적으로 보호하는 것이 목표다.

## 2. 낙관적 락 (Optimistic Lock)

엔티티에 버전 컬럼(`@Version`)을 두고, 갱신 시점에 읽은 버전과 현재 버전이 다르면 `OptimisticLockException`으로 충돌을 감지·롤백하는 방식.

| 장점 | 단점 |
| --- | --- |
| DB 락을 잡지 않아 평상시 처리량이 높다 | 충돌 시 예외 → **재시도 로직이 필수**, 경쟁이 심할수록 재시도가 폭증한다 |
| 데드락이 발생하지 않는다 | 인기 상품처럼 단일 행에 트래픽이 몰리면 재시도가 거의 매번 실패해 사실상 throughput이 무너진다 |
| 스키마 변경(버전 컬럼)만으로 적용, 구현이 단순 | **여러 엔티티에 걸친 불변식**(예: 정책 수량 차감 + 고객 쿠폰 생성 + 중복 검증)을 한 번에 보호하기 어렵다 |
| 읽기 위주·충돌이 드문 도메인에 적합 | 충돌 처리·재시도 한도·백오프를 직접 설계해야 한다 |

**판정**: 타임세일/쿠폰은 한정 수량에 트래픽이 집중되는 **고경쟁(high-contention)** 시나리오라 충돌이 상시 발생한다. 재시도 비용이 보호 이득을 상쇄하므로 부적합. (현재 `CouponPolicy` 등 엔티티에는 `@Version`을 두지 않았다 → 낙관적 락은 채택하지 않음.)

## 3. 비관적 락 (Pessimistic Lock)

`SELECT ... FOR UPDATE`(JPA `LockModeType.PESSIMISTIC_WRITE`)로 DB 행에 배타 락을 걸어, 트랜잭션이 끝날 때까지 다른 트랜잭션의 접근을 차단하는 방식.

| 장점 | 단점 |
| --- | --- |
| 충돌을 **사전 차단**하므로 재시도가 불필요하다 | 락 보유 동안 DB 커넥션을 점유 → **커넥션 풀 고갈** 위험 |
| 강한 일관성, read-modify-write 보호가 확실하다 | 락 범위·획득 순서가 어긋나면 **데드락** 가능 |
| 단일 DB 안에서는 구현이 직관적 | DB에 락 부하가 집중, 수평 확장이 어렵다 |
| | **DB 락은 데이터 행에만 걸린다.** "행이 아직 없는데 동시에 생성되는 중복 발급" 같은 케이스나, 락 경계를 DB 외부 자원과 맞추는 데 한계 |

**판정**: 단일 인스턴스·단일 DB라면 유효하지만, 락 대기가 곧 **DB 커넥션 점유**로 직결돼 부하가 DB에 쏠린다. 락 경계를 애플리케이션 레벨(요청 단위)에서 통제하고 싶은 본 프로젝트 요구와 맞지 않음.

## 4. Redis 분산 락 (Distributed Lock)

락 상태를 애플리케이션·DB 외부의 Redis에 두어, **여러 애플리케이션 인스턴스가 공유**하는 단일 락을 획득·해제하는 방식.

| 장점 | 단점 |
| --- | --- |
| 다중 인스턴스(scale-out)에서도 **전역 직렬화**가 가능하다 | Redis가 **단일 장애점(SPOF)**이 될 수 있다 → 가용성 설계 필요 |
| 락 부하를 DB에서 분리, DB 커넥션을 락 대기로 점유하지 않는다 | 락 보유 중 프로세스가 죽으면 락이 남는다 → **Lease Time(자동 만료)**으로 방어 |
| 락 키 단위로 보호 경계를 자유롭게 설계(`lock:timesale:{id}`, `lock:coupon:{id}`) | 네트워크 왕복 비용, 클럭/만료 기반이라 이론적 안전성 한계(Redlock 논쟁) |
| 락 획득/해제 라이프사이클을 **트랜잭션 바깥**에 둬 커밋-해제 순서를 명시적으로 통제 | TTL보다 임계영역이 길어지면 락이 먼저 풀릴 수 있어 임계영역을 짧게 유지해야 한다 |

**판정**: DB 부하 분리 + 다중 인스턴스 직렬화 + 락 경계의 유연성을 모두 만족하므로 **채택**. 단점(SPOF·락 잔류)은 Lease Time과 Fail-Closed 정책으로 방어한다.

## 5. Redisson Pub/Sub 방식 선택 근거

분산 락 구현 후보는 (a) Redis `SETNX` + 스핀 폴링 직접 구현과 (b) Redisson 라이브러리다. 본 프로젝트는 **Redisson**(`redisson-spring-boot-starter`)을 채택했다.

- **Pub/Sub 기반 대기**: 직접 구현의 `SETNX` 스핀 락은 락이 풀릴 때까지 짧은 간격으로 재조회(busy-wait)해 Redis 부하와 CPU를 낭비한다. Redisson은 락 해제 시 **Pub/Sub 채널로 통지**해 대기 스레드를 깨우므로, 폴링 없이 효율적으로 대기한다.
- **재진입(Reentrant) 락 + 소유권 검증**: Redisson `RLock`은 Lua 스크립트로 획득/해제를 원자적으로 처리하고, **현재 스레드가 소유한 락만 해제**(`isHeldByCurrentThread`)할 수 있어 "남의 락을 푸는" 오류를 막는다. 본 구현(`RedissonLockManager#unlock`)도 이 검증에 의존한다.
- **Lease 자동 만료 (TTL)**: TTL(Lease Time)을 명시해 락 보유 프로세스가 죽어도 자동 해제된다. 직접 구현 시 일일이 다뤄야 하는 만료를 라이브러리가 검증된 형태로 제공한다. 참고로 Redisson의 **Watchdog(자동 락 갱신)는 `leaseTime`을 지정하지 않을(`-1`) 때만** 동작하는데, 본 프로젝트는 `leaseTime=2초`를 **명시**하므로 Watchdog는 사용하지 않고 지정한 TTL 만료에 의존한다.
- **검증된 구현**: 분산 락의 까다로운 경계(원자적 해제, 만료, 통지)를 직접 구현하면 버그 위험이 크다. 표준 라이브러리 사용이 안전하다.

## 6. 본 프로젝트 적용 정책 (Wait / Lease / Fail-Closed)

선택한 Redisson 분산 락은 다음 정책으로 운영한다. 값과 보호 경계는 실제 구현과 일치한다.

| 항목 | 값 | 의미 / 근거 |
| --- | --- | --- |
| **Wait Time** | **3초** | 락 획득을 최대 3초 대기한다. 초과 시 더 기다리지 않고 즉시 실패 처리 → 요청이 무한정 매달려 스레드를 점유하는 것을 방지한다. |
| **Lease Time** | **2초** | 락 보유 최대 시간. 보유 프로세스가 죽어도 2초 뒤 자동 해제되어 락 잔류를 막는다. 임계영역(차감·생성)은 이보다 짧게 유지한다. |
| **Fail-Closed** | 락 획득 실패 시 **거절(503)** | 락을 못 잡으면(타임아웃·Redis 장애 포함) 보호 없이 진행하지 않고 `SERVICE_UNAVAILABLE`로 거절한다. 정합성(초과 판매/발급 방지)을 가용성보다 우선한다. |

> **용어 메모**: Wait Time 초과 시 더 기다리지 않고 즉시 거절하는 동작은 'Fail-Fast'적 성격이지만, 본 문서는 "보호 없이는 진행하지 않고 거절한다"는 정책 전체를 **Fail-Closed**로 통칭한다. 위키 Policies 섹션에는 'Fail-Fast'와 'Fail-Closed' 표현이 혼재하므로, 위키 용어를 'Fail-Closed'로 정렬하는 후속 작업을 고려한다.

**구현 매핑**

- 인터페이스: `LockManager` (`tryLock(key, waitTime, leaseTime, unit)` / `unlock(key)`), 기본값 `DEFAULT_WAIT_TIME=3s`, `DEFAULT_LEASE_TIME=2s`.
- 운영 구현체: `RedissonLockManager` (`@Primary`, `@Profile("!nolock | prod")`) — `RLock#tryLock`, 해제는 `isHeldByCurrentThread` 확인 후 `unlock`. 획득 중 예외/인터럽트 시 `false` 반환 → 상위에서 Fail-Closed.
- 테스트 대체: 동시성 테스트는 `LockManager`를 Mockito 목으로 대체해 락 동작(직렬화/no-op)을 제어한다. 운영·로컬·테스트 기본은 `RedissonLockManager`를 사용하고, 부하 측정 전용 `nolock & !prod` 프로파일에서만 `NoOpLockManager`(no-op)로 대체된다. `prod,nolock` 조합에서는 운영 안전을 위해 `RedissonLockManager`가 활성화된다.
- 타임세일: `TimeSaleOrderFacade` — 키 `lock:timesale:{timeSaleId}`, 획득 실패 시 503. **락 임계영역 안**에서 멱등성 검사(`checkAndSet`) + 수량 차감·주문 생성을 수행한다.
- 쿠폰: `CouponIssueFacade` — 키 `lock:coupon:{couponPolicyId}`, Wait 3s / Lease 2s, 획득 실패 시 503.

> **멱등성 검사 위치 차이(의도된 설계)**: 두 Facade는 멱등성 검사 시점이 다르다. `TimeSaleOrderFacade`는 **락 획득 이후**(임계영역 안)에서 `checkAndSet`을 호출하고, `CouponIssueFacade`는 **락 획득 이전**에 호출한다. 쿠폰 쪽은 동일 Request-ID의 중복 요청을 **락 대기 없이 먼저 빠르게 거절**(`COUPON_003`)하려는 의도다. 두 방식 모두 정합성에는 문제가 없으며, 멱등성 키 충돌 처리를 락 경합 앞단에 둘지 임계영역 안에 둘지의 트레이드오프(빠른 거절 vs 흐름 단순화) 차이다. Redis에 선점된 Request-ID는 완료/실패 상태 전환 없이 TTL(현재 10초) 동안 유지되므로, 비즈니스 처리 실패나 락 획득 실패 이후 같은 Request-ID 재시도도 정책상 중복 요청으로 차단된다.

**락 ↔ 트랜잭션 경계**: 락 획득/해제는 `@Transactional` 서비스 호출을 감싸는 **Facade(트랜잭션 바깥)**에 둔다. 서비스가 반환되면 커밋이 끝난 상태이므로 그 다음 `finally`에서 락을 해제한다. 락 해제가 커밋보다 먼저 일어나면 다음 대기 요청이 **미커밋 상태**를 보고 수량·중복 검증을 우회할 수 있기 때문이다.

## 7. 최종 비교 요약

| 기준 | 낙관적 락 | 비관적 락 | Redis 분산 락 (채택) |
| --- | --- | --- | --- |
| 충돌 처리 | 사후 감지 + 재시도 | 사전 차단(대기) | 사전 차단(대기) |
| 고경쟁 적합성 | 낮음(재시도 폭증) | 중간(DB 커넥션 점유) | 높음 |
| 부하 위치 | DB | DB | Redis(DB와 분리) |
| 다중 인스턴스 | 가능(행 단위) | 가능(행 단위) | 가능(전역 키 단위) |
| 다중 엔티티 불변식 보호 | 어려움 | 가능 | 가능(키로 경계 설계) |
| 주요 위험 | 재시도 비용 | 데드락·커넥션 고갈 | Redis SPOF·락 잔류 |
| 위험 완화 | 재시도 한도/백오프 | 락 순서/타임아웃 | **Lease Time·Fail-Closed** |

**결론**: 한정 수량에 트래픽이 집중되는 고경쟁 도메인에서, DB 부하를 분리하면서 다중 인스턴스 전역 직렬화와 유연한 락 경계를 동시에 만족하는 **Redisson Pub/Sub 분산 락**을 채택한다. 분산 락의 약점은 **Lease Time(자동 만료)**과 **Fail-Closed(정합성 우선 거절)** 정책으로 방어한다.
